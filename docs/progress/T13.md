#### Quyết định kỹ thuật
- **AuthUser stores itself in extensions:** Modified `AuthUser::from_request_parts` to `parts.extensions.insert(auth_user.clone())` after successful extraction. This allows downstream extractors (TenantContext, future ones) to access the authenticated user without requiring it as a handler parameter. This is a key pattern for extractor chaining.
- **TenantContext depends on AuthUser:** The extractor reads `AuthUser` from extensions first, then reads the header. If AuthUser is not found, returns `BadRequest("auth must be resolved before tenant context")`. Handler parameter order matters: `AuthUser` must come before `TenantContext`.
- **DB membership check:** Uses `SELECT EXISTS(SELECT 1 FROM tenant_members WHERE tenant_id = $1 AND user_id = $2)` for efficient single-row check. Returns 403 Forbidden if user is not a member.
- **Header name hardcoded as const:** `TENANT_HEADER = "x-tenant-id"` (lowercase). Config's `tenant_header` field is available but not yet wired to the extractor (will be done during integration in main.rs).
- **Rejection type is ApiError** (not AuthError), which allows proper 400/403 status codes for tenant-specific failures.

#### Blocker / Warning cho batch tiếp theo
⚠️ `PgPool` must be added to request extensions in main.rs for the DB membership check to work. Currently, the extractor will return `ApiError::Internal("database pool not configured")` if the pool is missing.
⚠️ The 3 tests use a stub lazy pool (no real DB). The membership check is not tested with a real database — that will be covered in T15 integration tests.
