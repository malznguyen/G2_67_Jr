#### Quyết định kỹ thuật
- **SharedConnection pattern:** `Arc<Mutex<PgConnection>>` wrapped in a `Clone` struct. The mutex guard is held briefly: middleware acquires lock, sets tenant, drops lock, handler locks to execute queries, drops lock, middleware re-locks to commit.
- **Transaction lifecycle:** Middleware manages `BEGIN` → `SET LOCAL app.tenant_id` → handler → `COMMIT`. The `SET LOCAL` is scoped to the transaction, so when the connection returns to the pool, the tenant setting is gone.
- **Connection detach:** Uses `PoolConnection::detach()` to get a raw `PgConnection` that can be managed independently from the pool. The connection is not returned to the pool until it's dropped (after COMMIT).
- **Error handling:** Middleware returns structured JSON errors (`rls-connection-failed`, `rls-missing-tenant`, `rls-missing-pool`, `rls-begin-failed`, `rls-set-tenant-failed`, `rls-commit-failed`) using the same envelope format.
- **Test strategy:** Tests use a stub lazy pool. Since there's no real DB, `pool.acquire()` fails, which validates the error path. Full integration testing with real DB is deferred to T15.

#### Blocker / Warning cho batch tiếp theo
⚠️ `SharedConnection` must be used by handlers for queries that need RLS context. Handlers that don't use `SharedConnection` (using `PgPool` directly) will NOT have the RLS context.
⚠️ The middleware must be layered AFTER TenantContext extraction (TenantContext must be in extensions before the middleware runs). In the axum router, this means `.layer(from_fn(rls_middleware))` must come before `.layer(Extension(tenant_ctx))`.
⚠️ Integration tests with real DB (T15) are required to verify the full RLS flow.
