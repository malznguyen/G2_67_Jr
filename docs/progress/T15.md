#### Quyết định kỹ thuật
- **FORCE ROW LEVEL SECURITY on tenants:** Required because the DATABASE_URL user (`gmrag`) is a superuser. Without `FORCE`, superusers bypass RLS. `FORCE` makes RLS apply even to table owners (but NOT superusers — that's why tests use `SET LOCAL ROLE gmrag_app`).
- **gmrag_current_tenant() in migrations:** The function was originally only in `infra/postgres/init.sql` (Docker entrypoint). Added to the identity_and_tenant migration so `sqlx::test` databases (which skip init.sql) have the function.
- **gmrag_app role in migration:** Added `CREATE ROLE gmrag_app NOLOGIN` + GRANTs to the identity_and_tenant migration. This ensures test databases have the role. The main Docker database already has it from init.sql.
- **SET LOCAL ROLE gmrag_app in tests:** Each test transaction switches to the non-superuser `gmrag_app` role before querying. This makes PostgreSQL enforce RLS policies, allowing us to verify isolation.
- **pgcrypto extension:** Added `CREATE EXTENSION IF NOT EXISTS "pgcrypto"` to the migration for `gen_random_uuid()` support in test databases.

#### Blocker / Warning cho batch tiếp theo
⚠️ The `identity_and_tenant` migration was modified after being applied to the main database. The checksum is now different. `sqlx migrate info` will show `different checksum`. This is harmless but should be noted. To fix: either revert the main database and re-run migrations, or accept the mismatch.
⚠️ RLS on `tenants` uses `FORCE ROW LEVEL SECURITY` — this means even table owners are subject to RLS. If a superuser needs direct access, they must use `SET ROLE` to a non-superuser first.
⚠️ Integration tests require `DATABASE_URL` env var pointing to a running PostgreSQL. Without it, `#[sqlx::test]` cannot create test databases.
