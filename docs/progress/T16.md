#### Quyết định kỹ thuật
- **Provision is optional in AuthUser:** Uses `if let Some(pool) = parts.extensions.get::<PgPool>()` — if PgPool is not in extensions, provisioning is silently skipped. This allows existing tests (which don't add PgPool) to continue working. In production, PgPool MUST be in extensions.
- **ON CONFLICT (id) DO UPDATE:** If the user already exists, updates `email` and `name` to match the latest claims. This handles profile changes in Keycloak gracefully.
- **Error handling:** If provisioning fails (DB error), returns `AuthError::UserNotFound("provision failed: ...")`. This is intentional — if we can't ensure the user exists in DB, auth should fail.
- **Test conflict with tenant tests:** The tenant tests originally added `PgPool` to extensions (for TenantContext's DB check). After T16, the `AuthUser` extractor also uses PgPool (for provisioning), causing the stub pool to fail on execute. Fixed by removing PgPool from tenant test setup — those tests only test header validation, not DB queries.

#### Blocker / Warning cho batch tiếp theo
⚠️ `PgPool` MUST be added to request extensions in main.rs for auto-provisioning to work in production. Without it, users will NOT be created in the DB.
⚠️ The `provision_user` function uses `ON CONFLICT (id) DO UPDATE` which means every auth request updates the user's email/name. This is acceptable for MVP but could be optimized later (e.g., only update if claims changed).
