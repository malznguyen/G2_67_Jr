#### Quyết định kỹ thuật
- **AuthError code format changed from integer to string:** `status.as_u16()` → `self.code()` returning kebab-case strings (`"invalid-token"`, `"missing-header"`, `"jwks-fetch-failed"`, `"user-not-found"`). This is a breaking change for any frontend consuming the old integer codes. All existing tests updated.
- **ApiError is the catch-all error type** for all non-auth error paths. It delegates to `AuthError` for auth failures via `From<AuthError>`, preserving the same envelope format.
- **`gmrag_core::Error` maps directly** to `ApiError::Core` with its existing kebab-case codes (`"database-error"`, `"config-error"`, etc.).
- **Envelope invariant enforced by test:** `envelope_always_has_error_object_with_code_and_message` iterates over every error variant and asserts the JSON shape `{ "error": { "code": <string>, "message": <string> } }`.
- **`ApiError.code()` returns `String`** (not `&str`) because it delegates to `AuthError.code()` for auth variants and `gmrag_core::Error.code()` for core variants, both returning `&'static str`, but we join them into a single return type.

#### Blocker / Warning cho batch tiếp theo
⚠️ `AuthError::IntoResponse` now uses string codes. Any downstream consumer (frontend) expecting integer codes will break. Frontend must update error parsing to handle string codes.
⚠️ `ApiError` is now available for all subsequent tasks (T13, T14, T16, T18) to use as handler return type.
