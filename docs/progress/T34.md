## Quyết định kỹ thuật
- **Reuse `gmrag_core::Config`** thay vì tạo `WorkerConfig` mới: `Config` đã có đầy đủ `S3Config`, `RedisConfig`, `OllamaConfig` (T5-T7). `S3Client::new(&cfg.s3)` (T35) và `RedisQueue::connect(&cfg.redis.url)` (T34) dùng trực tiếp. Tránh duplication, match pattern `gmrag-api`.
- **`JobQueue` trait + `MockQueue`**: redis crate không có built-in mock. Trait abstraction cho phép test poll loop mà không cần Redis thật. `MockQueue` dùng `Mutex<VecDeque>` — `pop_front()` = BRPOP, rỗng = timeout → `None`.
- **`MultiplexedConnection` thay vì `Connection`**: `redis::aio::Connection` deprecated trong 0.25+. `MultiplexedConnection` là recommended type. BRPOP với finite timeout (5s) an toàn trên multiplexed connection — worker không gửi concurrent commands khác.
- **`async_trait` crate**: Rust 1.75+ có native `async fn` trong traits, nhưng Send bound inference qua generic boundary (`poll_once<Q: JobQueue>`) phức tạp. `async_trait`_BOXING futures_giải quyết sạch sẽ, overhead negligible cho poll loop.
- **Lib+bin split**: `lib.rs` chứa tất cả logic + tests; `main.rs` chỉ `init_tracing()` + `gmrag_worker::run()`. Match pattern `gmrag-api` (bin + lib structure).
- **`tokio::select!` với `biased`**: `ctrl_c` được ưu tiên kiểm tra trước (`biased` keyword) — shutdown signal không bị block bởi BRPOP 5s timeout.
- **Pool rule DEFERRED**: T34 dùng `init_pool` (admin/gmrag) chỉ cho boot liveness check. `process_job` stub không ghi DB → không vi phạm RLS. Khi T37+ implement dual-write, **MUST** switch sang `init_app_pool` (gmrag_app role) + `SET LOCAL app.tenant_id = $job.tenant_id` per job.
